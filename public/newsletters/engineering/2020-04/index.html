Protobuf migration in progress

Ecosystem collaboration


A major upcoming (and long awaited) change in the Cosmos engineering stack is the migration from amino to protobuf. Tendermint Core is migrating to protobuf in the next major release. Cosmos-SDK users can continue using amino or transition to the protobuf compatible version. The process was broken into two sets:


  1. State encoding - completed

  2. Client encoding (tx generating, tx signing, and querying) - roughly 50% to 70% completed


The protobuf encoding primer that tracks the context and discussions related to this migration. The main goal with implementing an alternative encoder to amino is to facilitate easier integration for clients and Cosmos SDK users, while improving performance and cross-client/language compatibility. 


  • Github documentation on protobuf encoding proposal

  • Ongoing discussion on client encoding

  • Transaction client migration

  • Query migration

  • Cosmos SDK IBC module - part 1 protobuf migration and the part 1 and 2 combined issue detailing the work

  • Tendermint protobuf migration issue


⭐️ There will be FAQs and migration guides that will be done by the team to help you transition or evaluate the cost (labor and time) of transitioning to protobuf.

CosmWasm

Led by Confio


CosmWasm has recently made a proposal to customize their smart contract platform for the needs of the Cosmos Hub.


CosmWasm Cosmos Hub proposal in a nutshell


The Cosmos Hub integration of CosmWasm provides governance control to all aspects of a CosmWasm contract lifecycle and ensures compatibility with the hub. The Cosmos Hub governance mechanism would be able to control code upload, contract instantiation, upgrades, and destruction depending on what the Hub needs.


What it means


It provides a mechanism for governance to add new functionality to the Hub incrementally. This approach allows faster and easier iterations of the Cosmos Hub. The integration of CosmWasm in and of itself won’t add any functionality to the Hub, for example, it doesn't allow contracts to be uploaded or run by anyone.


4 reasons why Cosmos Hub needs CosmWasm integration


  1. After IBC (Inter-Blockchain Communication), there will be many competing hubs. The Cosmos Hub must provide necessary functionality in a timely manner to stay relevant
  2. "Rented security" is another large value proposition of the hub, and will likely require some custom logic for each chain that is added. CosmWasm would allow these rented security contracts to be added in a timely manner
  3. Adding code to Gaia can open up security issues in all other modules as well, so this is tightly controlled. A misbehaving CosmWasm contract cannot damage the functionality of other contracts
  4. This makes the barrier to entry to add functionality much lower


Upcoming development of CosmWasm


CosmWasm 0.8 release coming soon with three major points:

  • Intercontract/cross chain queries
  • Iterator to storage
  • Integration with native modules


Inter-Blockchain Communication integration


CosmWasm will begin a deep integration with IBC, allowing smart contracts to send and receive IBC packets, either with ICS20 or their own custom protocols.
