Led by Iqlusion, Regen Network, Vitwit, and Tendermint Inc
 

Stargate testnets

There are currently 3 stargate testnets running in Cosmos.

    • `cosmoshub-test-stargate-e` testnet, which simulates the process of upgrading the Cosmos Hub from the current `cosmoshub-3` to `cosmoshub-4` has gone through multiple iterations. Currently, the `cosmoshub-test-stargate-e` testnet is up and running. As Stargate introduces new command lines, REST architecture, and many more breaking changesâ€“it is crucial that validators have familiarity with the upgrade process. Check out the current state of the `cosmoshub-test-stargate-e` testnet via Mintscan and Big Dipper explorer.
    • `bigbang-2` testnet, a community-led testnet aimed at the wider Cosmos ecosystem and independent zone developers revolve around features and IBC composability testing, will undergo performance testing as well as the IBC proposal to enable IBC transfers. Check out the `bigbang-2` testnet on the Aneka explorer and the Big Dipper explorer.
    • `stargate-5` focuses on Gaia / Cosmos Hub, specifically for integration testing on legacy amino for service providers such as wallets, explorers, and exchanges a testnet oriented at wallets and explorers. Find more details here.
    • Ledger compatibility testing is underway. Governance vote tests and the test of Cosmovisor, which allows for a seamless software upgrade process for Cosmos SDK blockchains, on the Cosmos Hub will take place.
    • Major bugs have been identified and addressed. Minor bug fixes and patches will happen ahead of the Cosmos SDK RC5 release.
    • Read more about the Stargate upgrade and testnets here.

        Relayer in Rust

        Loads of integration work and fixes happened to help ibc-rs and to get us to better compatibility with Tendermint v0.34. We closed 45 issues (of varying degrees of complexity) in the past 2 months.

        • Ability to sign transactions in the relayer. Implemented logic in the relayer to allow submitting signed transactions to a stargate-4 chain. Transaction signing logic supports the new protobuf transaction encoding (as specified in Cosmos ADR-20).
        • Basic client, connection, and channel lifecycle in the relayer. Implemented transactions in the relayer for creating and updating clients, connection transactions (open-init, try, ack, confirm), as well as channel transactions (init, try, ack, confirm).
        • Implementation of chain runtime. Implement logic for a chain runtime. With this logic, the relayer must maintain a per-chain runtime called a `ChainRuntime`, which exposes a handle through which one can submit queries for retrieving a header or performing an on-chain query.
        • Light Client model-based testing and IBC audit. In the light client, we introduced a builder API for light client implementation and also started model-based testing on the light client. We continue to conduct model-based testing for other parts of Tendermint implementation in Go. You can see the latest status of model-based testing here. We also conducted an internal audit of IBC and opened a number of issues in the Cosmos SDK repo.
        • Relayer V0 master issue here.


        What's coming up for Relayer in Rust

        • Local chain support. We are working on a new type of Chain that is a local in-process chain. This chain will be our implementation of the IBC handlers. The local chain will also be used for in-process testing of the relayer. The local chain will allow us to run a blockchain (generating at least light blocks) and to have some on-chain state that can be queried and updated by transactions
        • Integration tests infrastructure. We are revamping some of the test infrastructure in our repository to allow better integration testing during our Continuous Integration (CI) process. We are leveraging Docker to spin up two chains (gaia) and a relayer, and automatically configure and run the relayer against these two chains. This will allow us to catch potential integration errors early and improve the quality of the relayer.
        • Improvements to light client integration and configuration. We are making improvements to the light client integration in the relayer as well as better support for the light client configuration in the relayer. This will improve the usability of the relayer and will translate into a better user experience.
